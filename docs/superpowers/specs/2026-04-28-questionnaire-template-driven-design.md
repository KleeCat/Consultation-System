# 问卷模板驱动动态渲染设计

## 1. 背景与目标

当前患者端问卷页存在三个问题：

1. 前端只写死了 2 道题，问诊信息不足。
2. 前后端各维护一份题目，后续扩题容易不一致。
3. 问卷交互仍是简单下拉表单，不符合正式问诊流程体验。

本次设计目标是把问卷能力升级为：

- 后端维护唯一题库，前端动态拉取并渲染。
- 问卷规模扩展到 12-15 题，本期先落地 13 题。
- 采用“按分组分页 + 单题单选卡片”的患者端交互。
- 保持现有提交流程基本稳定，同时增强后端校验与版本追踪。

---

## 2. 设计范围

### 2.1 本期纳入范围

- 问卷模板接口设计与实现
- 题库配置结构升级
- 13 道问卷题目与评分映射
- 前端问卷页动态渲染
- 三个分组的分页问答流程
- 必答校验、提交校验与错误提示
- 问卷版本号写入问诊会话

### 2.2 本期不纳入范围

- 多选题、填空题、量表滑杆
- 条件跳题和分支问答
- 问卷草稿自动保存到后端
- 中途退出后恢复答题进度
- 医生端在线维护题库

---

## 3. 总体方案

本次采用“后端题库驱动前端动态渲染”的方案。

### 3.1 方案原则

- 题目内容只保留一份来源，避免前后端双写。
- 展示配置与评分配置分离，降低维护耦合。
- 前端尽量复用现有提交流程，降低联动风险。
- 新增能力优先以可配置方式落地，方便后续继续加题。

### 3.2 核心思路

- `config/questionnaire.yaml` 负责定义问卷结构、分组、题目文案与选项。
- `config/question_scoring.yaml` 负责定义题目答案到体质分数的映射。
- 后端新增问卷模板读取接口，前端进入问卷页时动态拉取。
- 前端按分组展示题目，用户在本地完成所有选择后统一提交。

---

## 4. 题库配置设计

### 4.1 配置文件职责

#### `config/questionnaire.yaml`

负责前端展示与后端校验所需的静态模板信息：

- 问卷编码
- 版本号
- 问卷标题与说明
- 分组定义
- 题目定义
- 每题选项定义

#### `config/question_scoring.yaml`

负责问卷评分逻辑：

- 支持的体质维度
- 每道题每个选项的加分规则

### 4.2 问卷模板顶层结构

建议升级为以下结构：

- `questionnaire_code`
- `version`
- `title`
- `description`
- `groups`
- `questions`

### 4.3 分组结构

每个分组包含：

- `group_code`
- `group_title`
- `group_description`

本期推荐分为 3 组：

1. 体感与寒热
2. 饮食睡眠与排泄
3. 情绪压力与生活习惯

### 4.4 题目结构

每个题目包含：

- `question_code`
- `group_code`
- `question_text`
- `question_help`（可选）
- `required`
- `question_type`
- `options`

本期 `question_type` 固定为 `single_choice`。

### 4.5 选项结构

每个选项包含：

- `value`
- `label`
- `description`（可选）

这样可以支持每题自定义选项，而不是要求所有题目统一使用同一套 5 档频率。

## 5. 题目设计

本期先落地 13 题，覆盖气虚、阳虚、阴虚、痰湿、湿热、气郁与平和倾向。

### 5.1 第 1 组：体感与寒热

1. 最近精力状态怎么样？
2. 你平时是否怕冷、手脚发凉？
3. 你最近是否常觉得口干、咽干，想喝水？
4. 你身体是否容易觉得沉重、困倦、不清爽？

### 5.2 第 2 组：饮食睡眠与排泄

5. 你的胃口和饭后感受更接近哪种？
6. 你最近更常见哪种口感或口气状态？
7. 你的大便情况更接近哪种？
8. 最近睡眠情况如何？
9. 你最近出汗情况更接近哪种？

### 5.3 第 3 组：情绪压力与生活习惯

10. 最近情绪状态更接近哪种？
11. 压力大时你更容易出现哪种反应？
12. 你的作息更接近哪种？
13. 你的日常活动量更接近哪种？

### 5.4 评分映射原则

- 症状越明显，目标体质加分越高。
- 偏健康答案可给 `balanced` 少量加分。
- 少量题目允许同时影响两个维度，以提升问卷表达力。

例如：

- 饭后容易困重：偏向 `phlegm_dampness` 与 `qi_deficiency`
- 长期熬夜：偏向 `yin_deficiency` 与 `damp_heat`
- 情绪压抑、叹气明显：偏向 `qi_stagnation`

## 6. 后端设计

### 6.1 新增接口

新增接口：

- `GET /api/questionnaire/template`

返回内容包括：

- 问卷编码
- 版本号
- 标题与说明
- 分组列表
- 题目列表
- 每题选项列表

该接口仅用于前端动态渲染问卷，不依赖具体 `session_id`。

### 6.2 保留并增强提交接口

继续沿用现有接口：

- `POST /api/sessions/{session_id}/questionnaire`

提交格式保持为：

```json
{
  "answers": [
    {
      "question_code": "fatigue_level",
      "answer_value": "often"
    }
  ]
}
```

### 6.3 提交校验增强

后端提交时需要补强以下校验：

- `question_code` 必须存在
- `answer_value` 必须属于该题可选项
- 必答题不能缺失
- 同一题不能重复提交
- 不接受模板外的未知字段组合

若校验失败，应返回明确可读的错误信息，便于前端提示用户。

### 6.4 问卷版本追踪

提交成功后需要把当前模板 `version` 写入：

- `consultation_session.questionnaire_version`

这样后续即使题库升级，也能回溯历史问诊记录使用的是哪一版问卷。

## 7. 前端设计

### 7.1 页面目标

把当前静态下拉表单升级为更符合问诊流程的交互：

- 动态加载题库
- 按分组分页
- 单题单选卡片式作答
- 必答校验后再进入下一步
- 最后一组完成后统一提交

### 7.2 页面结构

问卷页建议拆成四个区域：

1. 顶部说明区
   - 标题
   - 简短说明
   - 当前进度
2. 当前分组区
   - 分组标题
   - 分组描述
3. 题目列表区
   - 单题卡片
   - 选项卡片
4. 底部操作区
   - 上一步
   - 下一步 / 提交并进入舌象采集

### 7.3 页面状态

前端需要维护以下状态：

- `template`
- `currentGroupIndex`
- `answers`
- `submitting`
- `loading`
- `loadError`
- `validationError`

### 7.4 交互流程

1. 进入问卷页后检查 `sessionId`
2. 请求后端问卷模板
3. 按 `group_code` 组装当前展示数据
4. 用户在当前组内作答
5. 未完成必答题时不允许进入下一组
6. 最后一组完成后调用现有提交接口
7. 成功后跳转到舌象采集引导页

### 7.5 组件拆分建议

建议将问卷页面拆成至少两个子组件：

- `QuestionnaireProgress`
- `QuestionCard`

如果后续样式继续增强，还可以继续抽离：

- `QuestionGroupHeader`
- `QuestionnaireFooterActions`

### 8.1 前端 API 扩展

在 `frontend/src/api/consultation.ts` 新增：

- `getQuestionnaireTemplate()`

同时定义对应的 TypeScript 类型：

- `QuestionnaireTemplateResponse`
- `QuestionnaireGroup`
- `QuestionnaireQuestion`
- `QuestionnaireOption`

### 8.2 路由与状态兼容

现有路由跳转关系基本不变：

- `profile` 完成后进入 `questionnaire`
- `questionnaire` 提交成功后进入 `capture-guide`

因此本期不需要更改整体患者端流程，只替换问卷页实现方式。

## 9. 测试设计

### 9.1 后端测试

至少覆盖以下场景：

1. 模板接口返回成功
2. 模板中包含版本、分组、题目与选项
3. 提交合法答案成功
4. 提交不存在的 `question_code` 失败
5. 提交非法 `answer_value` 失败
6. 必答题缺失时失败
7. 提交成功后写入 `questionnaire_version`

### 9.2 前端测试

至少覆盖以下场景：

1. 问卷模板动态渲染成功
2. 分组分页切换成功
3. 当前组未答完时不能进入下一步
4. 最后一组提交时 payload 正确
5. 模板加载失败时显示错误提示
6. 无 `sessionId` 时跳回欢迎页

### 9.3 手工回归

需要手工验证以下流程：

1. 基础信息页进入问卷页
2. 三组问卷依次完成
3. 最后一组提交成功
4. 进入舌象采集页
5. 后端分析流程继续可用

## 11. 风险与约束

### 11.1 主要风险

- 当前问卷页已有历史乱码，局部修补可能造成维护负担。
- 题库展示配置与评分配置若不同步，可能导致提交或评分异常。
- 前端动态渲染后，既有测试需要同步改造。

### 11.2 控制策略

- 问卷页优先做一次干净重构，不在旧静态实现上继续堆补丁。
- 题目文案与评分编码在同一轮变更中同步更新。
- 通过模板接口测试、提交校验测试和前端分页测试覆盖主要风险点。

---

## 12. 预期结果

完成本次改造后，系统将获得以下收益：

- 问卷题库改为后端唯一配置源
- 患者端问卷流程更完整、更接近真实问诊体验
- 后续加题、改文案、调选项只需改配置与评分规则
- 问卷版本具备可追溯能力
- 为后续继续扩展更多题型、条件跳题或后台题库管理打下基础
